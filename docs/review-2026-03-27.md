## Dimension 1: Accessibility (WCAG AA) — 42 / 100

### What's Working

- Form labels correctly associated via `htmlFor` (`token-name`, `edit-token-name`, etc.)
- Escape key closes modals and context menu (tested in unit tests)
- Camera has keyboard controls (WASD + arrow keys) in `CameraController.tsx`
- `DebuggerOverlay` correctly uses `aria-live="polite"` for dynamic updates
- `autoFocus` on primary inputs in `CreateTokenModal` and `EditTokenModal`
- `@storybook/addon-a11y` installed

### Gaps

#### Critical

**Modal/Dialog accessibility** — `Modal.tsx`
The modal is implemented as a `<div>` + `createPortal` without dialog semantics:

- No `<dialog>` element, no `role="dialog"`, no `aria-modal="true"`
- No focus trap — Tab/Shift+Tab escapes the modal freely
- Focus not returned to the trigger element on close
- No `aria-labelledby` linking the heading to the dialog
- Affects: `CreateTokenModal`, `EditTokenModal`, `ConfirmModal`

**Canvas content invisible to AT** — `Token.tsx`, `Board.tsx`
PixiJS tokens are rendered to WebGL canvas. Screen readers receive no information about:

- Which tokens are on the board
- Token names, sizes, or positions
- State changes (token added, moved, deleted)

No off-screen semantic alternative exists.

**Context menu has no menu semantics** — `ContextMenu.tsx`
Interactive items are plain `<button>` elements inside a `<div>`. Missing:

- `role="menu"` on container
- `role="menuitem"` on items
- No keyboard shortcut to open (e.g. Shift+F10 / application key)

#### High Priority

**Form errors not programmatically associated** — `CreateTokenModal.tsx`, `EditTokenModal.tsx`
React Hook Form / Zod validation errors display visually but:

- No `aria-invalid="true"` on inputs with errors
- No `aria-describedby` linking error messages to their inputs
- No `aria-required="true"` on required fields

**No semantic landmarks** — `__root.tsx`
Navigation and content areas use unsemantically styled `<div>` elements. Missing `<header>`, `<main>`, `<nav>` landmark regions.

#### Medium Priority

**Color contrast unverified**
The dark theme uses combinations that likely fail WCAG AA:

- `text-slate-500` on `bg-slate-900/80` — likely < 4.5:1
- `border-slate-400/45` (45% opacity borders) — very low contrast
- `bg-white/[0.05]` backgrounds — essentially invisible

**A11y testing not enforced**
`.storybook/preview.tsx` has `a11y: { test: "todo" }` — violations are noted but don't fail CI. No axe-core integration in Vitest test suite.

#### Low Priority

- Icon-only "+" button in `TokenLibraryOverlay.tsx` has no accessible label
- `TokenLibraryOverlay` panel and `DebuggerOverlay` panel have no `role="region"` + `aria-label`

### Recommendations

| Priority | Action                                                           | File(s)                                          |
| -------- | ---------------------------------------------------------------- | ------------------------------------------------ |
| P0       | Migrate `Modal` to `<dialog>` with `showModal()` + focus trap    | `Modal.tsx`                                      |
| P0       | Add `role="menu"` + `role="menuitem"` to `ContextMenu`           | `ContextMenu.tsx`                                |
| P1       | Add off-screen token list as AT alternative to canvas            | `Board.tsx` / `combat.tsx`                       |
| P1       | Add `aria-invalid`, `aria-describedby`, `aria-required` to forms | `CreateTokenModal.tsx`, `EditTokenModal.tsx`     |
| P1       | Add semantic landmarks to root layout                            | `__root.tsx`                                     |
| P2       | Audit color contrast; fix failing combinations                   | All components                                   |
| P2       | Set Storybook a11y to `"error"` mode                             | `.storybook/preview.tsx`                         |
| P3       | Add `aria-label` to icon-only buttons                            | `TokenLibraryOverlay.tsx`                        |
| P3       | Add `role="region"` + `aria-label` to overlay panels             | `TokenLibraryOverlay.tsx`, `DebuggerOverlay.tsx` |

## Dimension 2: Application Architecture — 82 / 100

### What's Working

- **Excellent layer separation**: React UI / PixiJS rendering / pure math utilities have clean boundaries
- **Minimal, focused stores**: Two Zustand stores with clear responsibilities (board placements vs. token library)
- **Immer middleware**: Immutable updates written as readable mutations
- **Factory functions**: `createToken()` is the single construction point for domain objects
- **Zod schemas**: Single source of truth for data shapes and validation
- **Camera context**: Separates render-cycle UI state from persistent domain state
- **Co-location balanced correctly**: Tests/stories adjacent to source; shared utilities centralized
- **No prop drilling**: Context and stores serve all cross-cutting data needs

### Gaps

#### Critical

**No Error Boundaries**
There are zero `<ErrorBoundary>` components in the tree. A WebGL context failure, a corrupt PixiJS texture, or an uncaught exception in the A\* pathfinder will crash the entire application. Given PixiJS complexity and external image loading, this is a significant reliability gap.

Critical wrapping points:

- Around `<Board>` — WebGL failures
- Around individual `<TokenDisplay>` — image/texture failures
- At the route level — unhandled promise rejections

#### High Priority

**Silent async failures** — `useTokenTexture.ts`
On image load error, `texture` is set to `null` with no logging, no user notification, and no visual indicator. Users see a blank token circle but receive no explanation.

#### Medium Priority

**`combat.tsx` accumulating UI state** — `src/routes/combat.tsx`
The main route currently orchestrates: modal open/close flags, deletion confirmation flow, context menu state, drag context, and library drop handler — approaching ~200 lines. Extracting a `useCombatUI()` hook would keep the route file as a thin composition root.

#### Low Priority

**`TokenSize` violates Open/Closed** — `types/token.ts`, `schemas/createToken.ts`
Adding a new size requires modifying: the type union, the Zod schema, the form `<select>` options, and any size-dependent logic. A size registry pattern (an exported array + `typeof SIZES[number]` type) would make this extendable without touching consumers.

### Recommendations

| Priority | Action                                             | File(s)                                            |
| -------- | -------------------------------------------------- | -------------------------------------------------- |
| P0       | Add Error Boundaries around Board and route        | New `ErrorBoundary.tsx`, `Board.tsx`, `combat.tsx` |
| P1       | Surface image load errors to users                 | `useTokenTexture.ts`, `Token.tsx`                  |
| P2       | Extract `useCombatUI()` from route                 | `combat.tsx`, new `hooks/useCombatUI.ts`           |
| P3       | Introduce size registry for extensible `TokenSize` | `types/token.ts`, `schemas/createToken.ts`         |

## Dimension 3: React Best Practices — 85 / 100

### What's Working

- **Hook design**: All 12 custom hooks follow single-responsibility, export explicit option/result types
- **Effect cleanup**: `useTokenTexture`, `Modal`, `CameraController`, `ContextMenu` all clean up correctly
- **Strict TypeScript**: `strict: true`, `noUnusedLocals`, `noUnusedParameters` all enforced
- **Schema-validated forms**: Zod + React Hook Form with inferred types (`z.infer<>`)
- **Justified memoization**: `useMemo` for A\* paths, PixiJS geometry, and object construction; `useCallback` for stable event handler references
- **Zustand per-value selectors**: Each component subscribes to only what it uses
- **Testing infrastructure**: 90% coverage target, `renderHook` + `act` patterns, co-located tests

### Gaps

#### Medium Priority

**`CameraController` handles three distinct input modes** — `CameraController.tsx`
At 176 lines, the component manages right-click pan, scroll-wheel zoom, and WASD/arrow keyboard navigation in a single file with interleaved effects and listeners. Each mode is an independent behavior that could be:

- Individually tested
- Individually enabled/disabled
- Composed differently for different input contexts (e.g., mobile)

Suggested extraction: `useCameraPan`, `useCameraZoom`, `useCameraKeyboard`.

**No `React.memo` on heavy presentational components**
`TokenDisplay` and `GridOverlay` re-render on every parent update, even when their props haven't changed. Both trigger PixiJS draw calls. `React.memo` with appropriate comparators would prevent spurious canvas redraws.

**`ContextMenu` `setTimeout(0)` pattern** — `ContextMenu.tsx:31-33`
The menu defers its outside-click listener by one tick to avoid immediately self-closing. This is order-dependent and fragile. A cleaner approach: track whether the menu was opened during the current event via a `useRef` flag, or use `stopPropagation` on the triggering element.

#### Low Priority

**No E2E tests**
Playwright is installed but unused. Unit tests cover logic well, but no test exercises a full user flow:

- Create token → drag to board → move token → delete token
- Camera pan/zoom interactions
- Modal open → form fill → submit → cancel

### Recommendations

| Priority | Action                                               | File(s)                           |
| -------- | ---------------------------------------------------- | --------------------------------- |
| P1       | Split `CameraController` into 3 focused hooks        | `CameraController.tsx`, new hooks |
| P2       | Add `React.memo` to `TokenDisplay` and `GridOverlay` | `Token.tsx`, `GridOverlay.tsx`    |
| P2       | Replace `setTimeout(0)` with ref-based open tracking | `ContextMenu.tsx`                 |
| P3       | Add Playwright E2E tests for critical flows          | New `e2e/` directory              |

## Dimension 4: VTT-Specific Excellence — 55 / 100

### What's Working

- localStorage persistence via Zustand `persist` middleware (both stores)
- A\* pathfinding for intelligent token movement along legal paths
- Smooth camera system with correct anchor math for zoom-at-cursor
- Token library with reusable templates separate from board state
- Locked tokens prevent accidental movement
- Drag-from-library-to-board workflow

### Gaps

#### Critical for a VTT

**No Undo/Redo**
Every action (add, move, delete, edit token) is permanent and irreversible. In a live combat session, accidental deletions or misdrops happen constantly. This is the single highest-impact missing feature.

Zustand's store shape is well-suited for a temporal state pattern. The `zundo` library wraps Zustand stores with an undo/redo stack and requires minimal changes to existing store code.

#### High Priority

**No persistence schema versioning** — `combatStore.ts`, `libraryStore.ts`
Both stores save to localStorage without a schema version. If `Token` gains a new required field, existing saved data will be missing it. There is no migration function and no detection of outdated data — silent corruption on first load after a deploy.

Fix: add `version` and `migrate` to the Zustand persist config.

#### Medium Priority

**No multi-select**
Tokens can only be interacted with one at a time. Group movement is a core VTT workflow (moving a pack of enemies, repositioning a party). The selection model needs:

- Click to select
- Shift+click to add/remove from selection
- Lasso drag to select by region
- Group move of selected tokens

**No session export/import**
Users cannot save a battle setup to a file or share it. JSON round-trip of the full store state is the minimum viable version. This enables backups, handoffs between GMs, and sharing pre-built encounters.

#### Low Priority

**No GM/player distinction**
All users have identical full-control permissions. A VTT needs at least a GM view (all tokens, full control) vs. player view (own token only, limited control).

**Real-time readiness**
The architecture doesn't block collaboration, but nothing is structured for it. State mutations are Immer-based and not serialized as discrete operations. Moving to an event-sourcing or CRDT model would need store restructuring.

### Recommendations

| Priority | Action                                           | Notes                                          |
| -------- | ------------------------------------------------ | ---------------------------------------------- |
| P0       | Add undo/redo to `combatStore`                   | Use `zundo` or temporal state pattern          |
| P1       | Add schema versioning + migration to both stores | Zustand `persist` `version` + `migrate` config |
| P2       | Implement token multi-select                     | New selection store + Shift+click + lasso      |
| P2       | Session export/import                            | JSON serialize full store state to file        |
| P3       | GM/player permission model                       | Design spike                                   |
